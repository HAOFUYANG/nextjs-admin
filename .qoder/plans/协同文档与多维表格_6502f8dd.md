# 协同文档与多维表格功能实施方案

## 一、对 plan.md 的分析与修正

### 1.1 合理的部分（保留）
- **Block 化统一模型**：文档、表格、评论统一抽象为 Block，长期方向正确。
- **Yjs + Socket.IO 协同**：行业标准方案，增量 update binary 同步效率高。
- **snapshot + incremental update 持久化**：PostgreSQL 存 snapshot + updates 表，恢复时 replay，设计正确。
- **分三阶段实施**：避免一次性过载，策略正确。
- **NestJS Gateway 做房间隔离**：`socket.join(table:${tableId})` 符合现有实时网关设计。

### 1.2 需要修正的部分

| plan.md 原设计 | 问题 | 修正方案 |
|---|---|---|
| Univer 作为表格渲染引擎 | Canvas 渲染与 Next.js SSR 兼容性差；API 迭代快、文档不完善；集成成本高 | **AG Grid Community**（DOM 渲染、MIT 协议、React 生态成熟） |
| 第一阶段同时做「文档+表格+颜色+评论+Presence」 | 范围过大，无明确边界 | 第一阶段聚焦「多维表格 CRUD + 基础协同 + 富文本文档 Block」；颜色/条件格式/Presence 放第二阶段 |
| 缺少具体数据库 Schema | 只有概念接口，无 Drizzle 表定义 | 补充完整的 `documents` / `doc_updates` / `workspaces` / `workspace_members` Drizzle Schema |
| 缺少 API 与 Socket 事件定义 | 无 REST 端点和 Socket 协议细节 | 补充完整的 CRUD API + `y-update` / `awareness` / `operation` 事件协议 |
| 权限直接到「单元格级」 | MVP 阶段过度设计 | 第一阶段只做「Workspace 成员 + 文档级读写权限」 |
| Tiptap + Yjs 集成无细节 | 未说明 `@tiptap/extension-collaboration` 如何配置 | 补充具体配置和 Yjs 文档绑定代码 |

### 1.3 技术选型结论

| 层级 | 选型 | 理由 |
|---|---|---|
| 前端框架 | Next.js 16（现有） | 无需变更 |
| 后端框架 | NestJS（现有） | 无需变更，复用现有 Socket Gateway |
| 数据库 | PostgreSQL + Drizzle（现有） | 无需变更 |
| 实时通讯 | Socket.IO + Yjs | 现有网关扩展 `y-update` 事件即可 |
| 富文本协同 | Tiptap 2 + `@tiptap/extension-collaboration` + y-prosemirror | 官方成熟方案 |
| 表格渲染 | `ag-grid-community` + `ag-grid-react` | DOM 渲染、SSR 友好、单元格编辑/排序/过滤社区版全支持 |
| 表格协同 | Yjs `Y.Map<recordId, Y.Map<fieldId, cellValue>>` | 细粒度单元格更新，与 AG Grid `rowData` 通过适配层转换 |
| 状态管理 | Zustand | 轻量，适合表格 UI 状态（排序、过滤、选中等） |
| 评论系统 | 自研（基于 Block 模型） | 评论也是一种 Block，挂在文档或记录上 |

## 二、数据库 Schema 设计（Drizzle）

### 2.1 核心表

```typescript
// apps/service/src/db/schema.ts

export const workspace = pgTable('workspace', {
  id: uuid('id').primaryKey().defaultRandom(),
  name: text('name').notNull(),
  ownerId: text('owner_id').notNull(), // 关联现有用户系统
  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
});

export const workspaceMember = pgTable('workspace_member', {
  workspaceId: uuid('workspace_id').notNull().references(() => workspace.id, { onDelete: 'cascade' }),
  userId: text('user_id').notNull(),
  role: text('role', { enum: ['owner', 'editor', 'viewer'] }).notNull().default('viewer'),
  joinedAt: timestamp('joined_at').defaultNow().notNull(),
}, (t) => [primaryKey({ columns: [t.workspaceId, t.userId] })]);

export const document = pgTable('document', {
  id: uuid('id').primaryKey().defaultRandom(),
  workspaceId: uuid('workspace_id').notNull().references(() => workspace.id, { onDelete: 'cascade' }),
  title: text('title').notNull().default('Untitled'),
  type: text('type', { enum: ['doc', 'table'] }).notNull(), // doc=富文本, table=多维表格
  snapshot: bytea('snapshot'), // Yjs state vector snapshot
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
  createdAt: timestamp('created_at').defaultNow().notNull(),
});

export const docUpdate = pgTable('doc_update', {
  id: uuid('id').primaryKey().defaultRandom(),
  documentId: uuid('document_id').notNull().references(() => document.id, { onDelete: 'cascade' }),
  updateData: bytea('update_data').notNull(), // Yjs update binary
  createdAt: timestamp('created_at').defaultNow().notNull(),
});

export const comment = pgTable('comment', {
  id: uuid('id').primaryKey().defaultRandom(),
  documentId: uuid('document_id').notNull().references(() => document.id, { onDelete: 'cascade' }),
  blockId: text('block_id').notNull(), // 关联的 Block ID
  userId: text('user_id').notNull(),
  content: text('content').notNull(),
  createdAt: timestamp('created_at').defaultNow().notNull(),
});
```

### 2.2 设计说明
- `document` 表统一存储「协同文档」和「多维表格」的元数据，通过 `type` 字段区分。
- `snapshot` 字段存 Yjs 文档的完整 state vector，用于快速恢复。
- `doc_update` 表存每次协同编辑产生的增量 update binary，用于 replay 恢复。
- `workspace` / `workspace_member` 实现基础的权限隔离。

### 3.2 Socket.IO 事件协议

```typescript
// 客户端 → 服务端
interface ClientEvents {
  'doc:join': (payload: { documentId: string }) => void;
  'doc:leave': (payload: { documentId: string }) => void;
  'doc:y-update': (payload: { documentId: string; update: Uint8Array }) => void;
  'doc:awareness': (payload: { documentId: string; awareness: Uint8Array }) => void;
  'doc:operation': (payload: { documentId: string; op: Operation }) => void;
}

// 服务端 → 客户端
interface ServerEvents {
  'doc:joined': (payload: { documentId: string; users: AwarenessUser[] }) => void;
  'doc:user-joined': (payload: { user: AwarenessUser }) => void;
  'doc:user-left': (payload: { userId: string }) => void;
  'doc:y-update': (payload: { documentId: string; update: Uint8Array }) => void;
  'doc:awareness': (payload: { documentId: string; awareness: Uint8Array }) => void;
  'doc:operation': (payload: { documentId: string; op: Operation }) => void;
}

interface AwarenessUser {
  id: string;
  name: string;
  color: string;
  selection?: { from: number; to: number } | { rowId: string; fieldId: string };
}

interface Operation {
  id: string;
  type: 'cell-update' | 'field-add' | 'field-update' | 'field-delete' | 'record-add' | 'record-delete' | 'view-config';
  payload: any;
  userId: string;
  createdAt: number;
}
```

## 五、AG Grid + Yjs 协同数据流

```
用户 A 编辑单元格
  → AG Grid onCellValueChanged
  → 获取 record Y.Map → set(fieldId, newValue)
  → Yjs 生成 update binary
  → socket.emit('doc:y-update', { documentId, update })
  → 服务端 Y.applyUpdate(serverDoc, update)
  → 服务端持久化 update 到 doc_update 表
  → 服务端 broadcast（排除发送者）
  → 用户 B 收到 update
  → Y.applyUpdate(localDoc, update)
  → Yjs observer 触发
  → 更新本地 rowData
  → AG Grid refreshCells()
```

## 七、风险与注意事项

1. **AG Grid Community 限制**：无 Row Grouping、Aggregation、Pivoting；若后续需要这些功能，需升级 Enterprise 或自研。
2. **Yjs 内存泄漏**：服务端需定期清理无客户端连接的 Y.Doc，或采用 LRU 缓存策略。
3. **snapshot 策略**：建议每 100 条 updates 或每 5 分钟生成一次 snapshot，避免恢复时 replay 过多。
4. **Next.js SSR**：AG Grid 和 Tiptap 都需在客户端渲染，页面组件需加 `"use client"` 或使用动态导入 `next/dynamic`。
5. **大型表格性能**：AG Grid Community 在处理 10万+ 行时性能下降，若数据量大需引入 Server-Side Row Model 或虚拟滚动。
