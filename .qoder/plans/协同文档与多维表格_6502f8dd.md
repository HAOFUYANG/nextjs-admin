## 二、阶段 E：Yjs + Socket.IO 协同架构（已落地）

### 2.1 架构决策

- **不引入 y-websocket**，复用现有 NestJS Socket.IO，在独立 namespace `/doc` 上自研 Yjs Provider，避免多起动一个 WS 服务。
- **服务端拥有权威 Y.Doc**：每个 documentId 在内存中保有一份 Y.Doc，从 DB snapshot + doc_update 重建。
- **增量 update 持久化 + 周期性 snapshot**：每 50 条 update 或房间清空时生成一次 snapshot，同时清理旧 update。
- **Awareness 不持久化**，仅服务端转发，由 y-protocols/awareness 自身超时机制清理。
- **二进制传输**：Socket.IO v4 天然支持 Buffer/ArrayBuffer，Uint8Array 未经 base64 直达。

### 2.2 事件协议（最终版）

```
client → server
  doc:join      { documentId, user: { name, color } }
  doc:leave     { documentId }
  doc:update    { documentId, update: Uint8Array }
  doc:awareness { documentId, awareness: Uint8Array }

server → client
  doc:sync      { documentId, state: Uint8Array }       // 初次全量
  doc:update    { documentId, update, from }            // 其他 peer 增量
  doc:awareness { documentId, awareness, from }         // 其他 peer 状态
  doc:peers     { documentId, peers: PeerInfo[] }       // 完整 peer 列表
  doc:peer-join { documentId, peer: PeerInfo }
  doc:peer-left { documentId, socketId }
```

### 2.3 后端实现

| 文件                                            | 职责                                                                      |
| ----------------------------------------------- | ------------------------------------------------------------------------- |
| `apps/service/src/document/doc-room.service.ts` | Y.Doc 内存池：hydrate / applyUpdate / snapshot / addClient / removeClient |
| `apps/service/src/document/doc.gateway.ts`      | namespace `/doc` 网关，处理 join/leave/update/awareness，peer 进出广播    |
| `apps/service/src/document/document.module.ts`  | 注册 DocRoomService + DocGateway                                          |

关键点：

- `applyUpdate` 后 fire-and-forget 写入 `doc_update`，不阻塞广播。
- `removeClient` 检测到房间清空时主动 snapshot 并释放 Y.Doc（避免内存泄漏）。
- Snapshot 完成后调用 `deleteUpdatesBefore` 清理旧 update，跨多轮 snapshot 保证 update 表不水象膨胀。

### 2.4 前端实现

| 文件                                                                 | 职责                                                                        |
| -------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| `apps/web/src/lib/yjs/YjsSocketProvider.ts`                          | 封装 Y.Doc + Awareness + Socket.IO，提供 `onPeersChange` / `onSyncedChange` |
| `apps/web/src/components/document/CollaborativeDoc/TiptapEditor.tsx` | Tiptap + `@tiptap/extension-collaboration`，顶栏展示同步状态 + Peer 头像    |

要点：

- StarterKit v3 必须禁用 `undoRedo: false`（以前的 `history: false` 在 v3 已重命名），否则与 Yjs Undo 冲突。
- 默认内容 seed 策略：`onSynced` 后判断 `peers.length === 1 && xmlFragment.length === 0`，仅首位独独进入的客户端 seed，避免重复插入。
- `origin === 'remote'` 标记避免“服端同步下来的 update 被再发出去”的环路。
- Awareness 可独立订阅，顶栏 `PeerAvatars` 子组件直接消费 `peers` 列表渲染为环形头像。

### 2.5 数据流

```
本地输入 → Tiptap → Y.XmlFragment → Y.Doc.update 事件
   → socket.emit('doc:update', { documentId, update })
   → 后端 Y.applyUpdate(serverDoc) + addUpdate 表
   → client.to(room).emit('doc:update', ...)
   → 其他客户端 Y.applyUpdate(localDoc, update, 'remote') → Tiptap 自动重渲染
```

### 2.6 迁移说明

- 原 `document.content`（HTML 字符串）在富文本场景下被 Yjs 取代，不再使用；表格场景阶段性保留其作为“本地 JSON 快照”，待 E6 迁移后一并废弃。
- 后端首次进入顶部以 `doc.snapshot + doc_update[]` 重建状态。如需从旧 content 一次性 seed，可以在 `DocRoomService.hydrate` 中补一段 fallback（本轮未加，避免覆盖多人编辑后的状态）。

## 三、阶段 E6：Canvas 表格 × Yjs（已落地，MVP 粗粒度）

### 3.1 目标与权衡

本轮以**最小代价跑通端到端表格协同**为目标，采用**粗粒度同步**策略：

- 把 `fields[]` 与 `records[]` 整体作为 JSON 字符串塞进 `Y.Map`。
- 任意一边变化 → 整体替换对侧。
- **代价**：多人并发改不同字段会以“最后写入者胜出”，丢失 CRDT 字段级合并。
- **收益**：复用现有 zustand store，零侵入；后续可平滑替换为 Y.Array<Y.Map> 细粒度结构。

### 3.2 Y.Doc 结构（MVP）

```ts
ydoc.getMap<string>("table");
// {
//   fields:  JSON.stringify(FieldConfig[]),
//   records: JSON.stringify(RecordData[]),
// }
```

### 3.3 同步桥 `apps/web/src/lib/yjs/table-yjs-bridge.ts`

- `bindTableStoreToYDoc(ydoc, { seedIfEmpty })`：双向绑定，返回 `{ unbind, hasRemoteData }`。
- 初始化逻辑：
  - 远程已有数据 → `loadData` 注入 store。
  - 否则若 `seedIfEmpty` 提供 → `loadData` + `tableMap.set(...)` 写回 ydoc 让其他 peer 共享。
- 远程→本地：`tableMap.observe`，过滤 `transaction.origin === 'local-store'` 防回环；通过 `setState({ fields, records })` **仅替换数据**而不重置 `selectedCell/editingCell` 等 UI 状态。
- 本地→远程：`useTableStore.subscribe((state, prev) => …)`，diff `state.fields` / `state.records` 引用变化，仅在真正变更时 `tableMap.set(..., LOCAL_ORIGIN)`。
- 用 `lastFieldsJson` / `lastRecordsJson` 字符串缓存做去重，避免 stringify 抖动产生空 update。

### 3.4 `CanvasTableView` 改造

- 删除 `fetchDocument/updateDocumentContent` 依赖。
- `useMemo([documentId])` 创建 `Y.Doc + YjsSocketProvider`。
- `useEffect` 订阅 `provider.onPeersChange` / `onSyncedChange`。
- 首次 `synced` 后调用 `bindTableStoreToYDoc`：当 `peers.length <= 1` 时把 `DEFAULT_TABLE_FIELDS / DEFAULT_TABLE_RECORDS` 作为 seed，多人进入时不 seed，避免覆盖远程。
- 卸载时 `unbind() + provider.destroy() + ydoc.destroy()`。
- 顶栏新增**同步状态指示**（已同步/连接中）+ **PeerAvatars**（与 Tiptap 顶栏同样的圆形头像组件，本地内联实现）。

### 3.5 数据流

```
本地 zustand mutation（addRecord/updateCell/...）
   → useTableStore.subscribe diff fields/records
   → tableMap.set('fields'|'records', JSON, origin='local-store')
   → ydoc.on('update') 发出 → socket.emit('doc:update')
   → 后端 Y.applyUpdate(serverDoc) + 持久化 + 广播
   → 其他客户端 Y.applyUpdate(localDoc, update, 'remote')
   → tableMap.observe 触发
   → useTableStore.setState({ fields, records })
   → React 组件重渲染
```

### 3.6 已知限制 & 下一步演进

- 粗粒度策略下，并发更新会冲突丢失。下一步演进路径已铺设好：
  1. 把 `tableMap` 拆为 `Y.Array<Y.Map>('fields') + Y.Map<Y.Map>('records')`（recordId → fieldId → value）；
  2. `table-store.ts` 的 mutations 改造为 `(payload, opts?: { fromRemote?: boolean })`，`fromRemote=true` 时不反向写 Y.Doc；
  3. observer 增量 diff 后调用对应 mutation 的 remote 版本；
  4. 视图状态（`sortConfig` / `filterConfigs` / `hiddenFields`）单独存入 `ydoc.getMap('tableMeta')`，仍走 LWW 语义。
- 列宽拖拽 / 排序筛选当前**不进入 Yjs**，仍是本地视图状态，符合“每人有自己的视图”预期。
- `DocRoomService.hydrate` 暂未做 `document.content` → Y.Doc 的 fallback seed；旧表格首次进入会以前端 DEFAULT*TABLE*\* 重新 seed。如需迁移历史数据，可在 hydrate 中加一段：当 ydoc 为空且 `doc.content` 含 `{fields,records}` 时 transactional 写入 `tableMap`。

## 一、对 plan.md 的分析与修正

### 1.1 合理的部分（保留）

- **Block 化统一模型**：文档、表格、评论统一抽象为 Block，长期方向正确。
- **Yjs + Socket.IO 协同**：行业标准方案，增量 update binary 同步效率高。
- **snapshot + incremental update 持久化**：PostgreSQL 存 snapshot + updates 表，恢复时 replay，设计正确。
- **分三阶段实施**：避免一次性过载，策略正确。
- **NestJS Gateway 做房间隔离**：`socket.join(table:${tableId})` 符合现有实时网关设计。

### 1.2 需要修正的部分

| plan.md 原设计                                 | 问题                                                                   | 修正方案                                                                                       |
| ---------------------------------------------- | ---------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Univer 作为表格渲染引擎                        | Canvas 渲染与 Next.js SSR 兼容性差；API 迭代快、文档不完善；集成成本高 | **AG Grid Community**（DOM 渲染、MIT 协议、React 生态成熟）                                    |
| 第一阶段同时做「文档+表格+颜色+评论+Presence」 | 范围过大，无明确边界                                                   | 第一阶段聚焦「多维表格 CRUD + 基础协同 + 富文本文档 Block」；颜色/条件格式/Presence 放第二阶段 |

## 二、数据库 Schema 设计（Drizzle）

### 2.1 核心表

```typescript
// apps/service/src/db/schema.ts

export const workspace = pgTable("workspace", {
  id: uuid("id").primaryKey().defaultRandom(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(), // 关联现有用户系统
  createdAt: timestamp("created_at").defaultNow().notNull(),
  updatedAt: timestamp("updated_at").defaultNow().notNull(),
});

export const workspaceMember = pgTable(
  "workspace_member",
  {
    workspaceId: uuid("workspace_id")
      .notNull()
      .references(() => workspace.id, { onDelete: "cascade" }),
    userId: text("user_id").notNull(),
    role: text("role", { enum: ["owner", "editor", "viewer"] })
      .notNull()
      .default("viewer"),
    joinedAt: timestamp("joined_at").defaultNow().notNull(),
  },
  (t) => [primaryKey({ columns: [t.workspaceId, t.userId] })],
);

export const document = pgTable("document", {
  id: uuid("id").primaryKey().defaultRandom(),
  workspaceId: uuid("workspace_id")
    .notNull()
    .references(() => workspace.id, { onDelete: "cascade" }),
  title: text("title").notNull().default("Untitled"),
  type: text("type", { enum: ["doc", "table"] }).notNull(), // doc=富文本, table=多维表格
  snapshot: bytea("snapshot"), // Yjs state vector snapshot
  updatedAt: timestamp("updated_at").defaultNow().notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});

export const docUpdate = pgTable("doc_update", {
  id: uuid("id").primaryKey().defaultRandom(),
  documentId: uuid("document_id")
    .notNull()
    .references(() => document.id, { onDelete: "cascade" }),
  updateData: bytea("update_data").notNull(), // Yjs update binary
  createdAt: timestamp("created_at").defaultNow().notNull(),
});

export const comment = pgTable("comment", {
  id: uuid("id").primaryKey().defaultRandom(),
  documentId: uuid("document_id")
    .notNull()
    .references(() => document.id, { onDelete: "cascade" }),
  blockId: text("block_id").notNull(), // 关联的 Block ID
  userId: text("user_id").notNull(),
  content: text("content").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

### 2.2 设计说明

- `document` 表统一存储「协同文档」和「多维表格」的元数据，通过 `type` 字段区分。
- `snapshot` 字段存 Yjs 文档的完整 state vector，用于快速恢复。
- `doc_update` 表存每次协同编辑产生的增量 update binary，用于 replay 恢复。
- `workspace` / `workspace_member` 实现基础的权限隔离。

### 3.2 Socket.IO 事件协议

```typescript
// 客户端 → 服务端
interface ClientEvents {
  "doc:join": (payload: { documentId: string }) => void;
  "doc:leave": (payload: { documentId: string }) => void;
  "doc:y-update": (payload: { documentId: string; update: Uint8Array }) => void;
  "doc:awareness": (payload: {
    documentId: string;
    awareness: Uint8Array;
  }) => void;
  "doc:operation": (payload: { documentId: string; op: Operation }) => void;
}

// 服务端 → 客户端
interface ServerEvents {
  "doc:joined": (payload: {
    documentId: string;
    users: AwarenessUser[];
  }) => void;
  "doc:user-joined": (payload: { user: AwarenessUser }) => void;
  "doc:user-left": (payload: { userId: string }) => void;
  "doc:y-update": (payload: { documentId: string; update: Uint8Array }) => void;
  "doc:awareness": (payload: {
    documentId: string;
    awareness: Uint8Array;
  }) => void;
  "doc:operation": (payload: { documentId: string; op: Operation }) => void;
}

interface AwarenessUser {
  id: string;
  name: string;
  color: string;
  selection?: { from: number; to: number } | { rowId: string; fieldId: string };
}

interface Operation {
  id: string;
  type:
    | "cell-update"
    | "field-add"
    | "field-update"
    | "field-delete"
    | "record-add"
    | "record-delete"
    | "view-config";
  payload: any;
  userId: string;
  createdAt: number;
}
```

## 五、AG Grid + Yjs 协同数据流

```
用户 A 编辑单元格
  → AG Grid onCellValueChanged
  → 获取 record Y.Map → set(fieldId, newValue)
  → Yjs 生成 update binary
  → socket.emit('doc:y-update', { documentId, update })
  → 服务端 Y.applyUpdate(serverDoc, update)
  → 服务端持久化 update 到 doc_update 表
  → 服务端 broadcast（排除发送者）
  → 用户 B 收到 update
  → Y.applyUpdate(localDoc, update)
  → Yjs observer 触发
  → 更新本地 rowData
  → AG Grid refreshCells()
```

## 七、风险与注意事项

1. **AG Grid Community 限制**：无 Row Grouping、Aggregation、Pivoting；若后续需要这些功能，需升级 Enterprise 或自研。
2. **Yjs 内存泄漏**：服务端需定期清理无客户端连接的 Y.Doc，或采用 LRU 缓存策略。
3. **snapshot 策略**：建议每 100 条 updates 或每 5 分钟生成一次 snapshot，避免恢复时 replay 过多。
4. **Next.js SSR**：AG Grid 和 Tiptap 都需在客户端渲染，页面组件需加 `"use client"` 或使用动态导入 `next/dynamic`。
5. **大型表格性能**：AG Grid Community 在处理 10万+ 行时性能下降，若数据量大需引入 Server-Side Row Model 或虚拟滚动。
